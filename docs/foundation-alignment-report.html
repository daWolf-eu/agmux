Verdict: the foundational spine is built faithfully and well. The non-optional core (protocol · store · hub · wrapper) plus the first consumers (cli · tui) are real and match the document closely. Everything still missing is an optional consumer or the distributed (B) tier — explicitly deferred by the doc, not designed out. The divergences are pragmatic substitutions (a TUI dash for the web dashboard, opentui for Ink) rather than departures from the vision.

Diverged

Went another way than the doc

Pragmatic substitutions
§12dashboard
Web dashboard → an interactive TUI “dash”
The doc's @agmux/dashboard is a web usage/metrics dashboard. No web package exists; instead a terminal dashboard (sortable/filterable table, live mirror + detail panes, kill/attach) ships inside the TUI. Same query API, different surface.
tui/src/opentui/DashApp.tsxcli/src/dash.ts
§12tui engine
TUI on opentui, not Ink
§12 specifies the TUI is built with Ink; the codebase refactored to @opentui/react as the sole implementation (Ink dash code dropped, per recent commits). Legacy Ink watch view lingers.
tui/src/opentui/run-manage.tsxgit log f05cb3c
§6 · §12store portability
No Postgres-portable interface yet — SQLite is hardcoded
The doc keeps the store “behind an interface that can swap to Postgres.” Today raw bun:sqlite SQL and SQLite-only pragmas are used directly, with no dialect abstraction. Reversible, but a real divergence from the (B)-ready stance.
store/src/index.ts · migrations.ts · schema.ts
§4 · §12launcher ergonomics
agmux run -p <name>, not agmux run <preset>
The doc's canonical invocation is a positional preset; the CLI uses a -p flag for profiles and treats a bare positional as an inline command. Minor surface divergence.
cli/src/run.ts · parse-run.ts

Bottom line

The foundation is honored where it matters most. Every standing principle in §14 holds in code: localhost-only, identity injected via AGMUX_SESSION_ID, event log as truth with rebuildable projections, one unified DB, additive schema, tmux as a first-class substrate, and agent_kind vs profile both first-class. The MVP slice is not just met — it's the floor the alpha already builds on.

What's missing is by design. comms, insights, multi-host, and delegation spawn are the optional/(B) layers the doc deliberately scoped out of (A); their seams (query API, parent_session_id, feed abstraction) are already in place.

Two divergences deserve a doc update, so the foundation stops describing a trajectory the code has left: the dashboard is a TUI surface (and the TUI runs on opentui, not Ink), and the store is not yet behind a Postgres-swappable interface — the one divergence that quietly weakens the (B) promise and is worth either restoring or re-scoping explicitly.
